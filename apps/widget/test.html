2. How the Security Works

🔒 Three-Layer Security:

Layer 1: API Key Exchange
The widget sends your API key to /widget/init endpoint
Backend validates the key and checks if the current domain is whitelisted

Layer 2: Session Token Issuance
If validation passes, a short-lived session token is issued
Token expires in 2 hours and is locked to the request domain

Layer 3: Secure Message Transmission
All chat messages use the session token (NOT the API key)
Session token is domain-locked and time-limited
If someone steals it: expires in max 2 hours, only works from original domain

3. Test Scenarios

✓ Success Case:

If this domain is whitelisted in your org settings, you'll see the widget loaded successfully with your custom branding (bot name, colors, etc.)

✗ Error Cases:

  • Invalid API Key → "Invalid or inactive API key"
  • Domain not whitelisted → "Domain not authorized"
  • Missing Origin header → "Origin or Referer header is required"
  • Expired session token → Auto-refreshes with new init

4. API Endpoints (Widget Security)

POST /api/v1/widget/init

Initialize session with API key (called once when widget loads)

{ "apiKey": "fpy_pub_xxxxx" } Response: { "sessionToken": "fpy_st_xxxxx", "expiresIn": 7200, "config": { "botName": "Fluxypy Bot", "primaryColor": "#6366F1", "welcomeMessage": "Hi! How can I help you today?" } }

POST /api/v1/widget/message

Send chat message (uses session token in header)

Headers: x-session-token: fpy_st_xxxxx Content-Type: application/json Body: { "message": "Hello!", "sessionId": "session_xyz" }
